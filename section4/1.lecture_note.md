# LangChain 강의 노트 — Section 4: ReAct Search Agent 구현

> **강의 주제**: LangChain으로 ReAct 검색 에이전트 구현 및 실행 흐름 이해
> **난이도**: 중급
> **사용 패키지**: `langchain`, `langchain-openai`, `langchain-tavily`, `langsmith`, `python-dotenv`

```python
# 전체 완성 코드
from dotenv import load_dotenv
load_dotenv()

from langchain import hub
from langchain.agents import AgentExecutor
from langchain.agents.react.agent import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_tavily import TavilySearch

tools = [TavilySearch()]
llm = ChatOpenAI(model="gpt-4", temperature=0)
react_prompt = hub.pull("hwchase17/react")

agent = create_react_agent(llm=llm, tools=tools, prompt=react_prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)
chain = agent_executor

def main():
    result = chain.invoke({
        "input": "search for 3 job postings for an ai engineer using langchain in the bay area on linkedin and list their details"
    })
    print(result["output"])

if __name__ == "__main__":
    main()
```

## 1. LangChain Tool — LLM이 외부 API를 사용하는 방법

LLM은 외부 API를 직접 호출할 수 없다. **Tool**은 LLM이 "어떤 외부 기능이 있고, 언제·어떻게 호출하면 되는지"를 알 수 있도록 스키마로 감싸는 래퍼 객체다.

```python
from langchain_tavily import TavilySearch

# TavilySearch는 LangChain Tool 인터페이스를 구현한 공식 통합 객체
tools = [TavilySearch()]

# Tool 내부 속성 확인
tool = tools[0]
print(tool.name)           # "tavily_search_results_json"
print(tool.description)    # "A search engine optimized for..."
print(tool.args)           # {"query": {...}, "include_domains": {...}, ...}
```

> 🔍 **내부 동작 원리**: Tool의 구조
>
> ```text
> Tool 객체가 LLM에 전달하는 정보:
>   ├─ name        : "tavily_search_results_json"  → LLM이 호출할 Tool 식별자
>   ├─ description : "A search engine optimized..." → LLM이 "언제" 사용할지 판단
>   └─ args schema : {query: str, include_domains: list, ...}
>                                                  → LLM이 "어떤 인자"로 호출할지 결정
>
> 실행 흐름:
>   LLM → "tool_name + args 결정" (추론만)
>   AgentExecutor → 실제 tool._run(args) 호출 (실행)
>   LLM은 API를 직접 호출하지 않는다.
> ```

> 💡 **실무 포인트**: Tool의 `description`과 `args` 설명이 명확할수록 LLM의 Tool 선택 정확도가 높아진다.
> 직접 Tool을 만들 때 모호한 설명은 잘못된 호출을 유발한다.
> **LangChain 공식 통합 패키지를 우선 활용하라** — 해당 서비스 팀이 LLM 친화적으로 설명을 직접 작성해두었다.

> ⚠️ **주의**: Python 함수를 `@tool` 데코레이터로 감싸 직접 Tool을 만들 수도 있다.
> 이 경우 `docstring`과 **타입 힌팅**이 LLM이 읽는 스키마가 되므로 반드시 명확하게 작성해야 한다.

🔗 [TavilySearch 통합 패키지](https://python.langchain.com/docs/integrations/tools/tavily_search/)

## 2. LLM 설정 — ChatOpenAI

```python
from langchain_openai import ChatOpenAI

# temperature=0: 매 실행마다 동일한 결과 보장 (결정론적)
llm = ChatOpenAI(model="gpt-4", temperature=0)
```

> 💡 **실무 포인트**: 에이전트 개발 시 `temperature=0`은 거의 필수다.
> 랜덤성이 있으면 동일한 질문에 다른 Tool을 선택하거나 다른 순서로 실행해
> 재현 불가능한 버그가 발생한다. 디버깅이 극도로 어려워진다.

> ⚠️ **GPT-4 vs GPT-3.5**: 이 강의에서 GPT-4를 사용하는 이유는 강의 후반부에 공개된다.
> ReAct 프롬프트 방식은 모델의 지시 이행 능력(instruction following)에 크게 의존하기 때문에
> 소형 모델일수록 동작이 불안정해질 수 있다.

🔗 [ChatOpenAI API 레퍼런스](https://python.langchain.com/api_reference/openai/chat_models/langchain_openai.chat_models.base.ChatOpenAI.html)

## 3. ReAct 프롬프트 — 에이전트 추론의 핵심

```python
from langchain import hub

# LangChain Hub에서 공개 ReAct 프롬프트 로드
react_prompt = hub.pull("hwchase17/react")

# 프롬프트 입력 변수 확인
print(react_prompt.input_variables)
# ['input', 'tools', 'tool_names', 'agent_scratchpad']
```

ReAct 프롬프트 템플릿 전문:

```text
Answer the following questions as best you can. You have access to the following tools:
{tools}

Use the following format:

Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question

Question: {input}
Thought: {agent_scratchpad}
```

| 변수 | 채우는 주체 | 내용 |
|------|-----------|------|
| `input` | 사용자 | 원래 질문 |
| `tools` | LangChain 자동 추출 | Tool 이름 + 설명 전체 |
| `tool_names` | LangChain 자동 추출 | Tool 이름 목록 |
| `agent_scratchpad` | AgentExecutor 관리 | Thought/Action/Observation 누적 기록 |

> 🔍 **내부 동작 원리**: ReAct 프롬프트가 에이전트를 만드는 방법
>
> ```text
> LLM은 이 포맷을 보고 반드시 다음 형식으로 응답한다:
>   Thought: "검색 결과가 필요하다"
>   Action: tavily_search_results_json
>   Action Input: "AI engineer LangChain Bay Area LinkedIn jobs"
>
> AgentExecutor가 이 텍스트를 파싱해:
>   1. Action → Tool 이름 추출
>   2. Action Input → Tool 인자 추출
>   3. Tool 실제 실행
>   4. 결과(Observation)를 agent_scratchpad에 추가
>   5. 다시 LLM 호출 → 반복
>
> "Final Answer:" 가 나타나면 → 루프 종료
> ```

> 💡 **실무 포인트**: ReAct 프롬프트는 Cursor, Claude Code, Devin 같은 현대 AI 에이전트 기술의 근간이다.
> 단순한 프롬프트 하나가 LLM에게 "도구를 쓰고 추론하는" 능력을 부여한다.
> 이 메커니즘을 이해하면 어떤 에이전트 프레임워크든 내부를 읽을 수 있다.

> ⚠️ **EU 엔드포인트 사용 시**: 공개 프롬프트는 **미국(US) LangSmith Hub**에만 호스팅된다.
> EU 인스턴스에서는 `hub.pull`이 실패한다.
>
> ```python
> # 해결 방법 1: 환경변수 제거
> import os
> os.environ.pop("LANGCHAIN_ENDPOINT", None)
>
> # 해결 방법 2: 프롬프트 직접 생성
> from langchain_core.prompts import PromptTemplate
> react_prompt = PromptTemplate.from_template("""...""")  # 위 템플릿 직접 복붙
> ```

## 4. `create_react_agent` — 추론 엔진 (Chain)

```python
from langchain.agents.react.agent import create_react_agent

# 추론 엔진 생성 — 이것은 Chain이지 Agent가 아니다
agent = create_react_agent(
    llm=llm,
    tools=tools,
    prompt=react_prompt,
)
# 반환값: Runnable (Chain)
```

> 🔍 **내부 동작 원리**: `create_react_agent` 내부 구조
>
> ```text
> create_react_agent가 만드는 Chain 파이프라인:
>
>   react_prompt
>     | (tools, tool_names, input, agent_scratchpad 변수를 채워 넣음)
>     ↓
>   llm.invoke()
>     | (LLM이 Thought/Action/Action Input 형식으로 응답)
>     ↓
>   OutputParser
>     | (텍스트에서 "Action: X", "Action Input: Y" 파싱)
>     ↓
>   AgentAction(tool="X", tool_input="Y") 또는 AgentFinish(output="...")
> ```

**`agent` vs `AgentExecutor` 비교**:

| | `agent` (create_react_agent 반환값) | `AgentExecutor` |
|--|-----------------------------------|-----------------|
| 타입 | Runnable (Chain) | AgentExecutor |
| 역할 | 추론만 담당 (다음에 뭘 할지 결정) | 실행 담당 (Tool 호출, 루프 관리) |
| Tool 실행 여부 | ❌ 하지 않음 | ✅ 직접 실행 |
| 루프 | ❌ 없음 | ✅ while 루프 |

> 💡 **실무 포인트**: **`agent`라는 변수명은 오해를 부른다.** 실제로는 단순한 LangChain Chain이다.
> "에이전트"라는 이름의 지능적인 동작(루프, Tool 실행)은 전부 `AgentExecutor`가 처리한다.
> 코드 리뷰 시 이 점을 팀원에게 반드시 설명해야 한다.

🔗 [create_react_agent API 레퍼런스](https://python.langchain.com/api_reference/langchain/agents/langchain.agents.react.agent.create_react_agent.html)

## 5. `AgentExecutor` — 실행 오케스트레이터

```python
from langchain.agents import AgentExecutor

agent_executor = AgentExecutor(
    agent=agent,      # 추론 체인 (create_react_agent 반환값)
    tools=tools,      # 실제 실행할 Tool 목록
    verbose=True,     # Thought/Action/Observation 로그 출력
)
chain = agent_executor  # 이후 LCEL 체인 연결을 위한 별칭
```

> 🔍 **내부 동작 원리**: AgentExecutor의 while 루프
>
> ```text
> while True:
>     # 1. 추론 엔진 호출 → 다음 행동 결정
>     output = agent.invoke({
>         "input": question,
>         "agent_scratchpad": scratchpad  # 지금까지의 Tool 호출 기록
>     })
>
>     if isinstance(output, AgentFinish):   # "Final Answer:" 감지
>         return output.return_values["output"]
>
>     if isinstance(output, AgentAction):   # Tool 호출 필요
>         # 2. Tool 실제 실행
>         observation = tools_dict[output.tool].run(output.tool_input)
>
>         # 3. scratchpad에 기록 추가
>         scratchpad += f"\nAction: {output.tool}\nAction Input: {output.tool_input}\nObservation: {observation}\n"
>
>         # 4. 다시 루프 → LLM 재호출
> ```

**실행 로그 예시** (`verbose=True`):

```text
> Entering new AgentExecutor chain...
Thought: I need to search for AI engineer job postings using LangChain in the Bay Area.
Action: tavily_search_results_json
Action Input: AI engineer LangChain Bay Area LinkedIn jobs

Observation: [{'url': 'https://...', 'content': 'Job 1: ...'}]

Thought: I now have enough job postings.
Final Answer: Here are 3 AI Engineer job postings...

> Finished chain.
```

> 💡 **실무 포인트**: LangSmith 트레이스를 함께 열어보면 실행 흐름이 훨씬 명확하게 보인다.
> 각 LLM 호출에서 실제로 어떤 프롬프트가 전송됐는지, agent_scratchpad에 무엇이 쌓였는지를
> 클릭 한 번으로 확인할 수 있다. **로컬 `verbose=True` 로그 + LangSmith 트레이스**를 함께 사용하라.

> ⚠️ **주의**: `chain.invoke(input={...})`는 동작하지 않는다.
> LangChain Runnable의 `invoke`는 **위치 인자**로 dict를 받아야 한다.
>
> ```python
> chain.invoke(input={"input": "..."})   # ❌ 키워드 인자 불가
> chain.invoke({"input": "..."})         # ✅
> ```

🔗 [AgentExecutor API 레퍼런스](https://python.langchain.com/api_reference/langchain/agents/langchain.agents.agent.AgentExecutor.html)

## 6. 실행 결과 구조 및 LangSmith 트레이스 분석

```python
result = chain.invoke({"input": "search for 3 AI engineer job postings..."})

# result는 dict
# result["input"]  → 원래 질문
# result["output"] → 최종 LLM 응답 (Final Answer)
print(result["output"])
```

**LangSmith 트레이스 구조**:

```text
AgentExecutor Run
  ├─ [1] PromptTemplate  → 입력 변수를 채워 완성된 프롬프트 확인 가능
  ├─ [2] LLM 호출        → Thought + Action + Action Input 응답
  ├─ [3] OutputParser    → 텍스트에서 AgentAction 파싱
  ├─ [4] Tool 실행       → tavily_search 호출 + 결과(Observation)
  ├─ [5] LLM 호출 (2차)  → scratchpad에 Observation 포함된 프롬프트로 재호출
  └─ [6] OutputParser    → "Final Answer:" 감지 → AgentFinish
```

> 🔍 **내부 동작 원리**: `agent_scratchpad`의 역할
>
> ```text
> 1차 LLM 호출 후 scratchpad:
>   ""  (비어있음)
>
> Tool 실행 후 scratchpad:
>   "Thought: I need to search...
>   Action: tavily_search_results_json
>   Action Input: AI engineer LangChain Bay Area
>   Observation: [{'url': '...', 'content': 'Job 1: ...'}]"
>
> 2차 LLM 호출 시 이 scratchpad 전체가 프롬프트에 삽입됨
> → LLM이 "나는 이미 검색을 했고 결과가 있다"는 것을 기억함
> → Final Answer 생성 가능
> ```

> 💡 **실무 포인트**: `agent_scratchpad`를 이해하면 에이전트가 왜 "기억"을 가진 것처럼 동작하는지 알 수 있다.
> 실제로는 매 LLM 호출마다 이전 모든 행동 기록을 프롬프트에 다시 넣는 방식이다.
> 이 때문에 Tool 호출이 많아질수록 프롬프트가 길어지고 **토큰 비용이 급증**한다.
> 프로덕션에서는 `max_iterations`와 적절한 Tool 설계로 호출 횟수를 최소화해야 한다.

🔗 [LangSmith 공식 문서](https://docs.smith.langchain.com/)

## 7. 다음 단계: Output Parser로 구조화된 출력 만들기

현재 `result["output"]`은 순수 텍스트다. 실제 소프트웨어에서는 Python 객체(dict, Pydantic 모델)로 받아야 한다.

```python
from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel

class JobPosting(BaseModel):
    title: str
    company: str
    location: str
    url: str

# Output Parser를 chain에 연결하면 LLM 응답을 자동으로 구조화
parser = JsonOutputParser(pydantic_object=JobPosting)
```

> 💡 **실무 포인트**: Output Parser도 내부적으로는 **프롬프트**다.
> Pydantic 스키마를 프롬프트에 포함시켜 LLM이 해당 형식의 JSON을 출력하도록 유도하고,
> 응답을 파싱해 Python 객체로 변환한다. 에이전트의 모든 것이 결국 프롬프트 엔지니어링이다.

🔗 [Output Parser 공식 문서](https://python.langchain.com/docs/concepts/output_parsers/)

## 핵심 요약

```text
✅ Tool = name + description + args schema 로 구성된 Python 함수 래퍼
✅ LLM은 Tool을 "결정"만 한다 — 실제 실행은 AgentExecutor가 담당
✅ description 품질 = Tool 선택 정확도 → 명확하게 작성할수록 에이전트가 정확해짐
✅ react_prompt = 에이전트 추론의 핵심 — Thought/Action/Observation 사이클 강제
✅ agent_scratchpad = Tool 호출 기록 누적본 — LLM이 "기억"을 갖는 방법
✅ create_react_agent = 추론 체인 (Chain), 실제 에이전트 동작이 아님
✅ AgentExecutor = while 루프 런타임 — Tool 실행 + scratchpad 관리 + 루프 종료 담당
✅ chain.invoke({"input": "..."}) — dict를 위치 인자로 전달 (키워드 인자 불가)
✅ verbose=True + LangSmith 트레이스로 실행 흐름을 시각적으로 확인
✅ 토큰 비용 주의: Tool 호출이 늘수록 scratchpad가 커지고 비용이 급증함
```
