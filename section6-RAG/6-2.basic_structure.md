## 1. LangChain Document Loaders (문서 로더)

LangChain이 이렇게 강력한 이유 중 하나는 수많은 주요 서드파티 데이터 소스(Google Drive, Notion, 로컬 파일 시스템 등)와 쉽게 연동할 수 있도록 도와주기 때문입니다. LangChain은 이 수많은 소스들과 통신할 수 있는 래퍼(Wrapper) 클래스들을 내장하고 있습니다.

로더를 통해 데이터를 가져오면, 데이터 소스가 무엇이든(텍스트, PDF, 파워포인트 등) LangChain은 이를 텍스트를 담고 있는 추상화된 객체인 즉 **`Document` (문서)** 객체로 변환합니다. 모든 데이터는 결국 텍스트 형태로 다뤄지며 이 `Document` 객체를 LLM으로 전송하게 됩니다.

```python
# TODO: 실제 사용 환경에 맞는 Loader로 대체 필요
from langchain_community.document_loaders import TextLoader

# 파일 시스템에서 텍스트 기반의 Document 객체를 불러오는 기본 예시
loader = TextLoader("harry_potter.txt")
documents = loader.load()
```

> 💡 **실무 포인트**: 현업에서는 데이터 소스가 매우 다양합니다(Confluence, Jira, Notion 등). LangChain Document Loader의 가장 큰 장점은 **데이터 원천이 무엇이든 동일한 `Document` 인터페이스로 데이터를 통일해 파이프라인 뒤쪽 코드를 재사용 가능하게 만들어준다는 점**입니다.

🔗 [LangChain Document Loaders 공식 문서](https://docs.langchain.com/oss/python/integrations/document_loaders)

## 2. Text Splitters (텍스트 분할기)

LLM의 컨텍스트 윈도우 한계(토큰 리밋)를 극복하기 위한 필수적인 단계가 바로 문서를 잘게 쪼개는 것입니다. 단순히 길이에 맞춰 자르는 것은 쉬워 보일 수 있으나, **의미론적 요소(Semantic Meaning)를 보존하면서 자르는 것**은 상당히 복잡한 문제입니다.

LangChain의 Text Splitter는 긴 텍스트를 의미가 단절되지 않는 선에서 적절한 크기의 청크(Chunk) 단위로 쪼개고, 필요시 이 청크들을 다시 조립할 수 있도록 돕습니다.

```python
# TODO: 공식 문서에서 실제 API 이름 및 최신 패키지 구조 확인 필요
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 재귀적으로 텍스트를 청크 단위로 자르는 객체 생성
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000, 
    chunk_overlap=200
)
chunks = text_splitter.split_documents(documents)
```

> 💡 **실무 포인트**: 텍스트 분할 방식(Split Strategy)은 RAG 성능, 즉 검색의 정확도를 좌우하는 핵심 파라미터입니다. 코드 문서, 마크다운 문서, 법률 문서 등 **문서의 도메인 특성에 맞춰 서로 다른 텍스트 스플리터를 적용**해야 좋은 성능을 기대할 수 있습니다.

🔗 [LangChain Text Splitters 공식 문서](https://docs.langchain.com/oss/python/integrations/splitters)

## 3. Embeddings (임베딩)과 Vector Space (벡터 공간)

방대하게 쪼개놓은 수백만 개의 텍스트 청크 중에서 LLM에게 전달할 "정확히 필요한 청크"만 어떻게 찾아낼 수 있을까요? 여기서 NLP의 클래식하면서도 유용한 기법인 **임베딩(Embeddings)**이 등장합니다.

임베딩 모델은 구문(텍스트, 이미지, 오디오 등)을 입력으로 받아 연속적인 고차원 공간(Vector Space) 상의 **숫자 배열(벡터, Vector)** 구조로 반환해 주는 블랙박스 역할을 합니다. 성능이 좋은 임베딩 모델일수록 **의미가 서로 비슷한 문장일 때 두 벡터 간의 거리(Distance)가 가깝게 위치하게 됩니다.**

- `Sentence 1`: "I want to order an extra large coffee."
- `Sentence 2`: "I'll have a tall coffee."
- `Sentence 3`: "Quiero pedir cafe extra grande." (다른 언어)
=> 이 세 문장은 사용 언어나 단어가 조금씩 달라도 **의미(Semantic)가 같으므로 임베딩 벡터 공간 안에서는 서로 매우 가깝게 배치**됩니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> 1. 원본 텍스트: "How tall is the Burj Khalifa?"
> 2. 임베딩 모델 통과: 블랙박스 연산
> 3. 벡터 출력: [0.012, -0.34, 0.98, ...] (수백~수천 차원의 숫자 배열)
> ```

🔗 [LangChain Text Embedding Models 공식 문서](https://docs.langchain.com/oss/python/integrations/text_embedding/)

## 4. Vector Databases (벡터 데이터베이스) 와 RAG 완성

임베딩으로 변환한 방대한 양의 벡터 데이터를 저장하고 관리하기 위한 전용 데이터베이스가 바로 **Vector Database(예: Pinecone)**입니다.

방대한 문서 전체의 청크들을 미리 임베딩해서 벡터 데이터베이스에 저장해 두면 검색 시 다음과 같이 동작합니다:

> 🔍 **내부 동작 원리**: 전체 RAG와 Vector Database 흐름도
>
> ```text
> [데이터 인덱싱 과정]
> 1. 거대 문서(수 GB) → Text Splitters → 수만 개의 텍스트 Chunk 배열
> 2. Chunk 배열 → Embedding Model → 수만 개의 Vector 배열
> 3. Vector 배열 → Vector Database (ex. Pinecone) 에 저장 (Persist)
> 
> [사용자 조회(Query) 과정]
> 4. User: "책에서 존이 앨리스한테 뭘 했어?" (Query 쿼리)
> 5. Query → Embedding Model → 쿼리 Vector로 변환
> 6. 쿼리 Vector를 Vector Database에 넘겨 "가장 거리가 가까운 임베딩 Vector(Nearest Neighbors)" 검색
> 7. 가장 일치도가 높은 Vector를 다시 원본 텍스트 Chunk로 변환하여 추출 (Relevant Chunks)
> 8. 프롬프트 증강: "컨텍스트: [Relevant Chunks]... 질문: 존이 앨리스에게 뭘 했어?"
> 9. LLM → 가장 빠르고, 정확하고, 토큰 낭비 없는 답변 반환 (Generation)
> ```

> 💡 **실무 포인트**:
> 일반적인 RDB(관계형 데이터베이스)에서의 검색은 '정확히 일치하는 문자' 위주의 쿼리인 반면 Vector Database의 검색은 **Similarity Search(유사도 검색)** 기반입니다. 거리가 얼마나 가까운지를 빠르게 계산해 주는 최적화 인덱싱 기술 덕분에 수백만 개의 문서를 매우 빠른 속도로 스캔하고 찾아낼 수 있습니다. 이로 인해 LLM 토큰 비용을 기하급수적으로 아끼면서 불필요한 API 요청을 없앨 수 있습니다.

🔗 [LangChain Vector Stores 공식 문서](https://docs.langchain.com/oss/python/integrations/vectorstores)

## 핵심 요약

```text
✅ Document Loader: 외부 데이터가 무엇이든 간에 LangChain이 읽을 수 있는 Document 객체로 통일화시켜 불러온다.
✅ Text Splitter: 길고 큰 문서를 의미(Semantic) 단위가 훼손되지 않게 최적화된 작은 조각(Chunk) 텍스트로 분할한다.
✅ Embeddings: 쪼개진 텍스트 청크를 임베딩 모델에 통과시켜 의미를 내포하는 고차원 숫자 배열(Vector)로 바꾼다.
✅ Vector Database: 변환한 벡터값들을 저장 및 색인하여, 사용자의 질문(Query 벡터)과 가장 유사한 내용 벡터를 매우 빠르게 계산하여 꺼내준다.
```
