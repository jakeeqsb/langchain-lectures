> **강의 주제**: 블랙박스였던 ReAct 에이전트의 내부 구조를 이해하고 밑바닥부터 직접 구현하기
> **난이도**: 고급
> **사용 패키지**: `langchain`, `langchain-openai`

## 1. LangChain 에이전트 내부를 직접 구현하는 이유

지금까지 에이전트는 `create_react_agent` 같은 함수 안에서 마법처럼 동작하는 '블랙박스(Black Box)'로 다루어졌습니다. 텍스트를 입력하면 알아서 적절한 도구를 찾아 실행하고 답변을 내놓았습니다.

이 섹션에서는 이 블랙박스를 해체하여 **ReAct 에이전트 알고리즘의 모든 구성 요소를 맨바닥부터(from scratch) 직접 구현**합니다.

> 💡 **실무 포인트**:
> LangChain 팀은 현재 LangGraph를 에이전트 개발의 표준으로 강하게 밀고 있습니다. 하지만 곧바로 LangGraph로 넘어가버리면 수많은 추상화(Abstraction) 때문에 제너레이티브 AI 에이전트의 본질을 놓치게 됩니다.
> 처음부터 직접 프롬프트를 짜고 실행 루프를 만들어보는 이 과정은 향후 LangGraph나 다른 프레임워크를 사용할 때 필요한 **근본적인 아키텍처 이해도**를 높여줍니다.

## 2. `@tool` 데코레이터 — 함수를 LangChain 도구로 변환

이전에는 `Tool` 클래스를 통해 수동으로 도구를 정의했지만, `@tool` 데코레이터를 사용하면 일반 파이썬 함수를 매우 쉽게 LangChain 도구로 변환할 수 있습니다.

```python
from langchain.tools import tool

# ⚠️ 공식 문서 확인 필요: 실제 사용되는 text_length 함수의 가상 구현체
@tool
def get_text_length(text: str) -> int:
    """텍스트의 문자 길이를 반환합니다."""
    # LLM이 불필요한 이스케이프 문자나 따옴표를 넣을 수 있으므로 정제
    cleaned_text = text.strip("'\n\"")
    return len(cleaned_text)

# 생성된 도구의 메타데이터 확인 (디버깅)
print(get_text_length.name)        # get_text_length
print(get_text_length.description) # 텍스트의 문자 길이를 반환합니다.
print(get_text_length.args)        # 인자 스키마 

tools = [get_text_length]
```

> 🔍 **내부 동작 원리**: `@tool` 데코레이터
>
> 1. 데코레이터는 함수의 이름(`name`), 인자(`args`), 반환 타입(`return`), 그리고 Docstring(`description`)을 추출합니다.
> 2. 이를 기반으로 LangChain 내부의 `StructuredTool` (또는 `Tool`) 객체를 자동 생성하여 반환합니다.
> 3. LLM은 이 메타데이터(`name`, `description`, `args`)를 읽고 "언제,어떤 인자로" 이 도구를 호출할지 추론합니다.

> 💡 **실무 포인트**: LLM이 도구에 인자를 전달할 때 줄바꿈(`\n`)이나 불필요한 따옴표 문자를 함께 넘기는 경우가 잦습니다. 실무에서는 함수 내부 최상단에 위 예시의 `cleaned_text`처럼 **문자열을 정제(Cleaning)하는 작은 방어 로직**을 넣어두는 것이 좋습니다.

🔗 [LangChain Tools 공식 문서](https://python.langchain.com/docs/concepts/tools/)

## 3. LangSmith Hub에서 ReAct 프롬프트 가져오기

에이전트 구현의 핵심은 도구를 선택하도록 설계된 **아주 특별한 프롬프트**입니다.
이 프롬프트는 LangChain 창시자인 Harrison Chase가 만들었으며, **LangChain Hub** (LangSmith의 프롬프트 마켓플레이스)에서 다운로드할 수 있습니다. (`hwchase17/react`)

### ReAct 프롬프트 템플릿의 핵심 구조

```text
Answer the following questions as best as you can. You have access to the following tools:
{tools}  <-- 도구 목록과 설명이 들어갈 자리

Use the following format:
Question: the input question
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]  <-- 도구 이름 목록 
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question

Question: {input}
Thought: {agent_scratchpad} <-- 도구 실행 기록이 차곡차곡 쌓일 자리
```

> 🔍 **내부 동작 원리**: **Chain of Thought + Few-Shot 메커니즘**
>
> 위 프롬프트는 사실 두 가지 강력한 프롬프트 엔지니어링 기법의 결합체입니다.
>
> 1. **Few-Shot**: `Question:` → `Thought:` → `Action:` 포맷을 명시적으로 보여주어, LLM이 무조건 이 형식에 맞춰 답변을 생성하도록 강제합니다.
> 2. **Chain of Thought**: `Thought:` (상황을 어떻게 해결할지 생각하는 단계)를 의도적으로 거치게 만들어, 모델의 추론 능력을 극대화합니다 (ReAct 논문 구현체).

## 4. 프롬프트 렌더링 (`PromptTemplate.partial`)

가져온 프롬프트에 우리가 만든 `tools` 정보를 주입해야 합니다.
`{input}`과 `{agent_scratchpad}`는 실행 시점마다 바뀌지만, **도구 정보(`{tools}`, `{tool_names}`)는 고정**되어 있으므로 `partial` 메서드를 써서 미리 채워둡니다.

```python
from langchain_core.prompts import PromptTemplate
from langchain.tools.render import render_text_description
from langchain import hub

# 1. 허브에서 프롬프트 다운로드 (또는 직접 텍스트로 정의)
react_prompt_template = hub.pull("hwchase17/react")

# 2. 도구 정보를 미리 채워넣기 (Partial)
prompt = react_prompt_template.partial(
    # 도구 객체 리스트를 텍스트 디스크립션으로 변환
    tools=render_text_description(tools),
    
    # 도구 이름만 콤마로 연결
    tool_names=", ".join([t.name for t in tools]) 
)
```

> 🔍 **내부 동작 원리**: `render_text_description` 함수
>
> LLM은 파이썬 객체를 이해하지 못하고 **오로지 텍스트만** 읽을 수 있습니다.
> `render_text_description(tools)`는 LangChain 유틸리티 함수로, 내부적으로 `tools` 리스트를 순회하며 `이름 - 설명 - 파라미터 스키마`를 한 줄의 예쁜 텍스트 포맷으로 렌더링해주는 역할만 수행합니다.

🔗 [PromptTemplate.partial 공식 문서](https://python.langchain.com/api_reference/core/prompts/langchain_core.prompts.prompt.PromptTemplate.html#langchain_core.prompts.prompt.PromptTemplate.partial)

## 5. LLM 추론 엔진 설정과 `stop` 토큰의 중요성

```python
from langchain_openai import ChatOpenAI

# 1. temperature=0 으로 설정 (창의성 배제, 일관된 결과)
llm = ChatOpenAI(
    model="gpt-3.5-turbo", # 대본 기준 모델, 실무에서는 gpt-4o 계열 권장
    temperature=0
).bind(
    # 2. 아주 중요한 stop 파라미터 지정
    stop=["\nObservation"]      
)
```

> 🔍 **내부 동작 원리**: `stop` 파라미터는 왜 필수인가?
>
> ReAct 루프의 포맷은 `Thought` → `Action` → `Action Input` → `Observation` 순서입니다.
> 여기서 **`Observation`(관찰 결과)은 LLM이 생성해서는 안 됩니다.** 파이썬 코드에서 실제 도구(예: 타빌리 검색, 길이 측정 함수 등)를 실행한 진짜 결과값이 들어와야 하는 자리입니다.
>
> 만약 `stop=["\nObservation"]` 이 없다면?
> LLM은 끝없이 글짓기를 이어가며 자신이 상상해낸 가짜 `Observation`(할루시네이션)을 스스로 적어버리고 그에 기반한 답변을 내놓게 됩니다. 이를 방지하기 위해 **"LLM아, 너는 딱 `Action Input` 까지만 적고 생성을 완전히 멈춰!"** 라고 강제하는 것이 바로 `stop` 토큰의 핵심 역할입니다.

> 💡 **실무 포인트**:
> 다양한 LLM 벤더(Anthropic, Gemini, OpenAI 등)마다 `stop` 파라미터를 넘겨받는 방식이나 민감도가 다릅니다.
> 위 예시 코드처럼 `bind(stop=[...])` 로 공통 인터페이스를 사용하는 것이 `langchain_core` 최신 버전의 권장 방식입니다.
> *참고: 만약 파싱 에러(Parsing Error)가 자꾸 발생한다면 십중팔구 프롬프트 템플릿의 들여쓰기(Indentation) 오류나 이 `stop` 토큰 배열이 잘못 입력된 경우입니다.*
