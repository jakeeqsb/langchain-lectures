# Section 6 — RAG: LCEL 없는 수동 Retrieval 아키텍처 (Naive RAG)

> **강의 주제**: 정통 RAG의 5단계(Retrieve → Format → Prompt → LLM → Parse)를 수동(Manual) 코드로 바닥부터 구현하며 내부 프로세스와 LangSmith 디버깅 한계 이해하기
> **난이도**: 중급 / 고급
> **사용 패키지**: `langchain-core`, `langchain-openai`, `langchain-pinecone`

## 1. Vector Store Retriever — 연관 문서를 찾아오는 사냥개

> **"왜 벡터 저장소(Vector Store) 자체를 안 쓰고 Retriever로 변환시킬까요?" (Why)**
> Pinecone, Chroma, FAISS 같은 벡터 데이터베이스들은 저마다 데이터를 찾고(`similarity_search`) 필터링하는 파이썬 코드가 완전히 다릅니다. LangChain은 이 수많은 DB 벤더들의 파편화된 검색 함수들을 `BaseRetriever`라는 하나의 표준 인터페이스로 통일(추상화)시켰습니다. 덕분에 `vectorstore.as_retriever()` 한 줄만 호출하면, DB 종류가 바뀌어도 똑같은 `.invoke()` 메서드 하나로 관련된 문서 조각들을 똑같이 찾아올 수 있습니다.

작성한 코드에서는 Pinecone 벡터 DB 인스턴스를 생성한 뒤, 가장 유사도가 높은 문서 3개(`k=3`)를 가져오는 Retriever(검색기)로 변환해 사용합니다.

```python
from langchain_openai import OpenAIEmbeddings
from langchain_pinecone import PineconeVectorStore

embeddings = OpenAIEmbeddings()

# 1. Pinecone DB 연결
vectorstore = PineconeVectorStore(
    index_name=os.environ["INDEX_NAME"], 
    embedding=embeddings
)

# 2. 검색기로 변환 (코사인 유사도 기반 상위 3개 문서 추출)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# 데이터 검색 실행
docs = retriever.invoke("What is Pinecone in machine learning?")
```

> 💡 **실무 포인트**:
>
> - **Retriever는 단지 껍데기일 뿐입니다.** `as_retriever()`를 호출한다고 해서 Pinecone DB의 모든 데이터가 메모리로 올라오는 것이 아닙니다. 단순히 `invoke()`가 불릴 때 Pinecone SDK의 `query` API를 대신 때려주는(Proxy) 가벼운 래퍼(Wrapper) 클래스입니다.
> - **k값 튜닝(검색 정확도 vs 비용/오답률)**: 현업에서는 `search_kwargs={"k": 5}` 처럼 가져올 문서 조각(Chunk)의 개수를 정밀하게 튜닝해야 합니다. 너무 적게 가져오면 정답 정보가 부족하고, 너무 많이 가져오면 토큰 비용이 폭발하며 연관성 없는 데이터(Noise)로 인해 LLM이 오답(Hallucination)을 내뱉기 쉽습니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> [사용자 질문 (invoke)] ──▶ (내부적으로 Embeddings 모델 호출)
>                                        │
>     ┌──────────────────────────────────┘
>     ▼
> [Pinecone 벡터 DB] ──▶ (코사인 수식으로 가장 거리가 가까운 벡터 색인)
>     │
>     ▼
> [반환: LangChain Document 클래스 객체 3개 묶음 (k=3)]
> ```
>
> `retriever.invoke()`가 실행되는 순간, 내부적으로는 사용자의 텍스트 질문을 또 다른 OpenAI 임베딩 API를 사용해 '1536배열 벡터'로 숫자로 치환한 뒤 Pinecone으로 쿼리를 전송합니다.

## 2. Naive RAG 파이프라인 수동 구현 (Without LCEL)

> **"LCEL(LangChain Expression Language)이라는 최신 문법이 있는데, 왜 굳이 하나하나 촌스럽게 수동으로 코드를 짤까요?" (Why)**
> RAG 시스템이 "어떤 문서를 가져오고, 그걸 문자열로 ఎలా 합치고, 프롬프트의 어느 구멍에 끼워 넣는지"에 대한 **로우 레벨 블랙박스(Blackbox)를 뜯어보기 위해서**입니다. 이 절차적 단계를 완벽하게 체화하고 넘어가야 추후 LCEL 매직 체인(`retriever | format | prompt | llm`)을 짰을 때 오류가 발생하면, 어느 구간에서 데이터 형태가 꼬였는지 파악하는 스나이퍼급 디버깅이 가능해집니다.

일반적인 RAG의 5단계(Retrieve ➜ Format ➜ Prompting ➜ LLM ➜ Parse)를 Python의 일반적인 절차적 코드로 나열한 모습입니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

llm = ChatOpenAI()

prompt_template = ChatPromptTemplate.from_template(
    """Answer the question based only on the following context:
{context}

Question: {question}
Provide a detailed answer:"""
)

def retrieval_chain_without_lcel(query: str):
    # Step 1: Retriever가 연관 문서 3개 추출 (List[Document])
    docs = retriever.invoke(query)

    # Step 2: 3개의 문서 덩어리를 "\n\n"으로 이어 붙여서 하나의 거대한 문자열로 합침 (Formatting)
    context = "\n\n".join(doc.page_content for doc in docs)

    # Step 3: Prompt 템플릿의 빈 칸({context}, {question})에 값 주입
    messages = prompt_template.format_messages(context=context, question=query)

    # Step 4: 완성된 프롬프트를 LLM(GPT)에 전송 (결과는 AIMessage 객체)
    response = llm.invoke(messages)

    # Step 5: 최종 텍스트 내용만 추출 (Parse)
    return response.content
```

> 💡 **실무 포인트**:
> **수동 파이프라인(Legacy)의 치명적 한계**
> 현업의 프로덕션 환경(예: 수천 명이 쓰는 챗봇 API)에서 위와 같이 일반 파이썬 함수로 체인을 짜면 심각한 문제가 발생합니다.
>
> 1. **비동기(Async) 처리 부재**: 한 사람의 검색이 끝날 때까지 서버가 `invoke`에 매달려있어 대기줄(Block)이 발생합니다.
> 2. **실시간 스트리밍(Streaming) 부재**: 타자 치듯 글자가 실시간으로 출력되는 제너레이터(Generator) 구현이 매우 까다롭습니다.
> 3. **Observability(LangSmith 디버깅) 박살**: 함수로 분리해두면 LangSmith 대시보드에서 `Retriever` 동작 따로, `LLM` 동작 따로, 시분할 쪼개기 로그가 남아서 하나의 '요청 흐름(Trace)'으로 묶어서 에러를 추적하기 매우 힘듭니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> query: "What is Pinecone?"
>        │
> 1. [Retriever] ➔ [Doc("Pinecone은 DB"), Doc("서버리스다"), Doc("머신러닝용")]
>        │
> 2. [Formatter] ➔ 거대한 하나의 String 결합 ("Pinecone은 DB\n\n서버리스다\n\n...")
>        │
> 3. [Prompt]    ➔ 지시어 + Context 융합 (ChatPromptTemplate)
>        │
> 4. [LLM]       ➔ GPT 모델 연산 대기 (병목 지점)
>        │
> 5. [Result]    ➔ "Pinecone is a vector db..." 반환
> ```

🔗 [ChatPromptTemplate 공식 문서](https://reference.langchain.com/python/langchain-core/prompts/chat/ChatPromptTemplate), [LCEL / Runnables 공식 레퍼런스](https://reference.langchain.com/v0.3/python/core/runnables/langchain_core.runnables.base.Runnable.html)

## 3. RAG가 없을 때의 Hallucination (환각 현상)

> **"RAG를 적용하기 전과 후는 실제로 데이터베이스 지식적으로 어떤 차이가 나나요?" (Why)**
> GPT-3.5와 같은 구형 모델이거나 프리-트레이닝 기간이 끝난(21년도 지식 단절 등) 모델들은 Pinecone과 같이 신규 급상승한 기술에 대한 데이터가 뇌(Weights)에 없습니다. RAG 파이프라인 없이 그냥 질문을 던지면 **모르는 걸 아는 것처럼 천연덕스럽게 꾸며내는 심각한 환각(Hallucination)** 오류를 범합니다.

```python
from langchain_core.messages import HumanMessage

query = "what is Pinecone in machine learning?"

# RAG 미적용 (원시 LLM 호출)
result_raw = llm.invoke([HumanMessage(content=query)])
print(result_raw.content)
# 출력 (환각): "Pinecone algorithm은 모델 파라미터 최적화를 검색하는 기법입니다..." (완전한 오답)

# RAG 적용 (우리가 만든 수동 함수 호출)
result_without_lcel = retrieval_chain_without_lcel(query)
print(result_without_lcel)
# 출력 (정답): "Pinecone은 클라우드 기반 관리형 벡터 데이터베이스입니다..."
```

> 💡 **실무 포인트**:
>
> - **환각의 위험성**: 금융, 의료, 사내 보안 규정 등을 다루는 인공지능이 "Pinecone은 하이퍼파라미터 알고리즘입니다" 같은 오답을 내뱉을 경우 심각한 법적 책임을 물을 수 있습니다. RAG의 본질은 모델의 천재성(IQ)과 외부 데이터베이스의 지식(Dataset)을 완전 분리하여 팩트 기반(Grounded) 답변을 강제하는 데에 있습니다.

## 핵심 요약

```text
✅ Vector Store Retriever: `as_retriever()`는 온갖 종류의 벡터 DB의 검색 함수를 `.invoke()` 하나로 통일시키는 강력한 LangChain의 추상화 인터페이스입니다.
✅ 파이프라인 5단계: Retrieve(문서 획득) ➔ Format(문자열 조합) ➔ Prompt(지시어 융합) ➔ LLM(생성) ➔ Parse(결과 필터링)의 뼈대를 갖습니다.
✅ 수동 구현의 한계: 이 로직들을 일반 파이썬 함수로 짜면 비동기(Async) 처리, Streaming, 그리고 LangSmith에서의 통합 디버깅(Trace Tracker)을 날로 먹을 수 없습니다. (이를 해결하기 위해 LCEL이 등장함)
✅ RAG의 의의: 파인튜닝(추가 학습) 없이도 환각(Hallucination) 현상을 차단하고 완벽하게 최신 데이터 기반(Grounded)의 정답을 도출합니다.
```
