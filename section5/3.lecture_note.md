## 1. LLM의 생각 흐름 추적하기 (`AgentCallbackHandler`)

에이전트가 어떤 생각(`Thought`)을 하고, 어떤 결과(`Observation`)를 받고 있는지 투명하게 보려면 LangChain의 콜백(Callbacks) 시스템을 이용해 LLM의 API 호출 전/후를 로깅(Logging)해야 합니다.

### 예시 코드 — 커스텀 콜백 핸들러 작성 및 주입

```python
from langchain_core.callbacks import BaseCallbackHandler
from langchain_openai import ChatOpenAI

# 1. BaseCallbackHandler를 상속받아 커스텀 핸들러 생성
class AgentCallbackHandler(BaseCallbackHandler):
    # LLM에 프롬프트가 전송되기 직전에 트리거
    def on_llm_start(self, serialized: dict, prompts: list[str], **kwargs) -> None:
        print("\n[LLM 전송 프롬프트]")
        # 첫 번째 프롬프트 텍스트 전문 출력
        print(prompts[0])
        print("="*50)

    # LLM에서 응답이 도착한 직후에 트리거
    def on_llm_end(self, response, **kwargs) -> None:
        print("\n[LLM 응답 결과]")
        # 생성된 첫 번째답변 텍스트 출력
        print(response.generations[0][0].text)
        print("="*50)

# 2. LLM 초기화 시 콜백 핸들러 주입
llm = ChatOpenAI(
    model="gpt-3.5-turbo", # 실무에서는 gpt-4o 권장
    temperature=0,
    callbacks=[AgentCallbackHandler()] # ← 모든 LLM 호출 시 로그가 찍힙니다.
).bind(stop=["\nObservation"])
```

> 🔍 **내부 동작 원리**: Callbacks 이벤트 루프
>
> LangChain 내부의 객체(LLM, Chain, Agent, Tool)들은 실행될 때 발생하는 주요 이벤트마다 특정 메서드(`on_llm_start`, `on_tool_end` 등)를 호출(Callback)하도록 설계되어 있습니다. 이 이벤트 핸들러를 오버라이드하면, 디버거 없이도 어느 시점에 파이프라인의 어떤 데이터가 오가는지 실시간으로 추적(Tracing)할 수 있습니다.

> 💡 **실무 포인트**:
> 과거에는 이러한 커스텀 Callback 핸들러를 직접 짜서 로그를 터미널 콘솔에 출력해야 했습니다. 현재는 이 모든 콜백의 내부 흐름을 시각화해 주는 툴이 바로 **LangSmith**입니다. 다만 위처럼 BaseCallbackHandler의 구조를 알면, 특정 이벤트 타임에 슬랙(Slack) 알림을 보내거나 데이터베이스에 로그를 적재하는 등 독자적인 로깅 시스템을 구축할 수 있습니다.

🔗 [LangChain Callbacks 공식 문서](https://python.langchain.com/docs/concepts/callbacks/)

## 2. LLM Playground에서 직접 확인하는 프롬프트 조작

대본에서 강사는 터미널에 찍힌 프롬프트 이력(`on_llm_start` 로그)을 복사해, **OpenAI Playground**에 직접 붙여 넣어 실행해 보라고 권장합니다.
이는 LLM이 프롬프트를 인식하는 방식을 가장 날것(Raw)으로 확인할 수 있는 방법입니다.

> ⚠️ **공식 문서 확인 필요**: Playground는 OpenAI 플랫폼의 웹 UI이므로 [OpenAI Playground 페이지](https://platform.openai.com/playground)에서 직접 확인해야 합니다.

* **Playground 실행 시나리오**:
  1. 찍힌 로그(도구 목록 + ReAct 템플릿 포함)를 붙여넣고 LLM을 돌린다.
  2. `stop` 토큰 설정을 실수로 빠뜨린 채로 돌리면, LLM이 `Thought` → `Action` → `Observation` → `Thought` 순서로 **끝도 없이 혼자 대답을 이어가는(할루시네이션) 모습**을 직접 눈으로 목격할 수 있습니다.
  3. UI에서 `Stop sequences` 설정창에 `\nObservation`을 추가하고 다시 돌리면, 딱 `Action Input`까지만 작성하고 마법처럼 작문을 멈춥니다.

## 3. 완전한 에이전트: 최종 While 루프

이제 [Part 1]의 도구 설정, 프롬프트 생성 모듈과 [Part 2]의 히스토리 내역(`intermediate_steps`) 관리, 그리고 방금 전에 추가한 로그 핸들러까지 모두 조합하여 **순수 파이썬 While 루프** 하나로 묶습니다.

이 코드가 바로 `AgentExecutor`의 내부 실체입니다.

```python
from langchain_core.agents import AgentFinish

# 히스토리 초기화

while not isinstance(agent_step, AgentFinish):

    # 1. 포맷팅된 텍스트 히스토리를 프롬프트에 주입 (Part 2 참고)
    agent_scratchpad_text = format_log_to_str(intermediate_steps)

    # 2. 파서를 통해 AgentAction 혹은 AgentFinish로 변환
    agent_step = output_parser.parse(llm_response)

    # 3. 만약 도구 실행이 필요하다면 (AgentAction인 경우)
    if not isinstance(agent_step, AgentFinish):

        tool_name = agent_step.tool

# While 루프 탈출 후: 최종 답 출력
print("최종 답:", agent_step.return_values["output"])
```

> 🔍 **내부 동작 원리**: `AgentExecutor`
> 이 `while` 블록 코드가 바로 도구 실행 대행자인 `AgentExecutor`의 내부 구현체입니다. `create_react_agent`가 추론 체인(`prompt | llm | parser`)을 만들면, 그 체인 밖에서 무한 루프를 돌리며 툴을 실행하고 다시 체인에 피드백을 넣어주는 껍데기가 바로 `AgentExecutor`입니다.
