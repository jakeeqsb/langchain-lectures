> **강의 주제**: ReAct 루프 완성하기 — 기억의 누적(`agent_scratchpad`)과 종료 조건(`AgentFinish`)
> **난이도**: 고급
> **사용 패키지**: `langchain`, `langchain-core`

## 1. 에이전트의 단기 기억 장치: `agent_scratchpad`

ReAct 루프가 1회 실행된 후, 도구(Tool)를 실행해서 얻은 결과(Observation)를 받게 됩니다. 이때 이 결과를 그냥 넘기고 다음 질문을 하면 LLM은 방금 자신이 도구를 실행했다는 사실을 잊어버리고 같은 도구를 반복해서 호출하는 무한 루프에 빠집니다.

이를 방지하기 위해 **지금까지의 모든 생각과 행동의 역사를 기록해 두는 공간**이 필요한데, 이것이 바로 프롬프트에 정의된 `{agent_scratchpad}` 입니다.

### 예시 코드 — 이력 저장 및 전달

```python
from langchain.agents.format_scratchpad import format_log_to_str

# 1. 루프 전: 히스토리를 저장할 빈 리스트 생성
intermediate_steps = []

# --- ReAct (while) 루프 시작 ---
# ... (LLM 추론, 도구 선택, 도구 실행 후 observation 획득 과정 생략) ...

# 2. 루프 끝: 에이전트의 선택(agent_step)과 그 결과(observation)를 튜플로 저장
intermediate_steps.append((agent_step, observation))

# 3. 다음 루프 시작 전: 파이썬 객체를 LLM이 읽을 수 있는 텍스트로 변환
agent_scratchpad_text = format_log_to_str(intermediate_steps)

# 4. 프롬프트에 전달할 때 포맷팅된 텍스트 리턴
prompt_variables = {
    "input": "사용자 질문",
    "agent_scratchpad": agent_scratchpad_text 
}
```

> 🔍 **내부 동작 원리**: `format_log_to_str`
>
> `intermediate_steps` 리스트 안에는 `AgentAction` (LangChain 전용 파이썬 객체)과 문자열 결과값이 들어있습니다. LLM은 파이썬 객체를 이해할 수 없습니다.
> `format_log_to_str` 유틸리티 함수는 이 객체들을 까서 아래와 같은 순수 문자열 텍스트 덩어리로 예쁘게 변환해 줍니다.
>
> ```text
> "Action: get_text_length\nAction Input: 'hello'\nObservation: 5\n"
> ```
>
> 이렇게 변환된 텍스트가 다음 루프의 프롬프트 꼬리(`agent_scratchpad`)에 착착 쌓이면서 에이전트는 중복된 행동을 피할 수 있게 됩니다.

> 💡 **실무 포인트**: 파싱 에러(Parsing Error) 해결의 90%
> 대본에서 언급되었듯, LLM이 `Final Answer`와 `Action`을 동시에 출력해서 파싱 에러가 나는 경우가 정말 많습니다. 이는 `agent_scratchpad` 내부 텍스트의 끝부분에 불필요한 공백이나 줄바꿈이 들어갔기 때문일 확률이 매우 높습니다.
> 프롬프트 템플릿의 들여쓰기와 공백 하나하나가 파싱 에러의 주범이 될 수 있으므로, 에러가 난다면 프롬프트의 공백을 가장 먼저 의심하세요.

🔗 [Agent Scratchpad 및 Log Format 공식 문서](https://python.langchain.com/docs/concepts/agents/#agent-scratchpad)

## 2. 루프의 종료: `AgentAction` vs `AgentFinish`

Output Parser가 LLM의 텍스트 답변을 파싱했을 때 반환하는 객체는 딱 두 가지뿐입니다.

1. **`AgentAction`**: 에이전트가 *"아직 답을 모르겠으니 툴을 한 번 더 써볼게"* 라고 판단했을 때 반환됩니다. (도구 이름과 인자 포함)
2. **`AgentFinish`**: 에이전트가 *"이제 최종 답을 알아냈어!"* 라고 판단했을 때 반환됩니다.

### 예시 코드 — 분기 처리 (루프 탈출)

```python
from langchain_core.agents import AgentFinish

# ... LLM 출력 결과 파싱 ...
agent_step = output_parser.parse(llm_response_text)

# AgentFinish가 반환되었다면? -> while 루프 종료!
if isinstance(agent_step, AgentFinish):
    # return_values 딕셔너리의 "output" 키에 최종 답변이 담겨있음
    final_answer = agent_step.return_values["output"]
    print("최종 답변:", final_answer)
    break  # 루프 탈출
else:
    # AgentAction이 반환되었다면 -> 도구 실행 후 intermediate_steps 첨부 (루프 계속)
    tool_name = agent_step.tool
    # ... 도구 실행 로직 생략 ... 
```

> 🔍 **내부 동작 원리**: Output Parser는 어떻게 `AgentFinish`를 아는가?
>
> LLM이 생성한 텍스트 중에 **`Thought: I now know the final answer`** 와 **`Final Answer:`** 라는 문자열 패턴이 발견되면, 파서는 더 이상 도구를 호출할 필요가 없다고 판단하고 해당 문자열 뒤의 내용을 통째로 잘라서 `AgentFinish` 객체의 `return_values["output"]` 에 담아 반환합니다.

> 💡 **실무 포인트**: Agent 디버깅 패턴
> 실무에서 에이전트를 만들 때는 반드시 `isinstance` 분기점이나 `try-except` 블록에 디버그 로그(또는 print문)를 심어두세요. LLM이 `AgentAction`도 아니고 `AgentFinish` 포맷도 아닌 이상한 텍스트를 뱉을 때가 있는데, 이때 파서가 박살나며 앱이 죽어버리는 것을 방지하기 위한 안전장치(`OutputFixingParser` 등)가 필수적입니다.

🔗 [AgentAction 및 AgentFinish API 문서](https://python.langchain.com/api_reference/core/agents/langchain_core.agents.AgentAction.html)

## 핵심 요약

```text
✅ observation(툴 실행 결과) 자체만으로는 다음 행동을 결정할 수 없으며, 반드시 모든 history가 필요하다.
✅ intermediate_steps 리스트는 에이전트의 이전 기억을 담는 임시 저장소다.
✅ format_log_to_str 함수는 파이썬 객체인 히스토리를 LLM이 읽을 수 있는 텍스트로 치환하여 agent_scratchpad에 넣어준다.
✅ 파싱 에러가 발생한다면 프롬프트 끝의 띄어쓰기, 줄바꿈, 들여쓰기 문제일 확률이 매우 높다.
✅ Output Parser의 파싱 결과가 AgentFinish 타입일 때가 바로 에이전트 While 루프가 종료(break)되는 시점이다.
```
