## 1. Pydantic 스키마 설계 — LLM 출력의 설계도

Pydantic은 Python 타입 힌팅을 사용하는 데이터 검증 라이브러리로, LLM 구조화 출력에 특히 효과적이다.

```python
# schemas.py
from typing import List
from pydantic import BaseModel, Field

class Source(BaseModel):
    """에이전트가 사용한 출처 소스 스키마"""
    url: str = Field(description="출처 URL")

class AgentResponse(BaseModel):
    """에이전트의 최종 응답 스키마 (답변 + 출처 목록)"""
    answer: str = Field(description="에이전트의 최종 답변")
    sources: List[Source] = Field(
        description="답변 생성에 사용한 출처 목록",
        default_factory=list   # 인스턴스마다 독립적인 빈 리스트 생성
    )
```

> 🔍 **내부 동작 원리**: Pydantic이 LLM에 전달하는 정보
>
> ```text
> AgentResponse 클래스 정의 시 Pydantic이 자동으로 JSON 스키마 생성:
>   {
>     "title": "AgentResponse",
>     "description": "에이전트의 최종 응답 스키마...",
>     "properties": {
>       "answer": {"type": "string", "description": "에이전트의 최종 답변"},
>       "sources": {
>         "type": "array",
>         "items": {"$ref": "#/definitions/Source"},
>         "description": "답변 생성에 사용한 출처 목록"
>       }
>     }
>   }
>
> 이 스키마가 프롬프트 또는 Function Calling 방식으로 LLM에 전달됨
> → LLM은 이 형식에 맞춰 JSON을 생성
> ```

> 💡 **실무 포인트**: `Field(description="...")` 의 설명 품질이 LLM의 출력 정확도를 결정한다.
> 특히 **중첩 구조(nested objects)** 에서 각 필드의 역할을 명확히 기술해야
> LLM이 올바른 위치에 올바른 데이터를 채워 넣는다.

> 💡 **실무 포인트**: `default_factory=list`를 사용하는 이유 — `default=[]`처럼 mutable 기본값을
> 직접 쓰면 모든 인스턴스가 동일한 리스트 객체를 공유하는 Python의 유명한 버그가 발생한다.
> `default_factory=list`는 인스턴스 생성마다 새 리스트를 만들어 이를 방지한다.

🔗 [Pydantic 공식 문서](https://docs.pydantic.dev/latest/)  
🔗 [LangChain Pydantic 통합](https://python.langchain.com/docs/concepts/structured_outputs/#schema-definition-pydantic-vs-typeddict-vs-json-schema)

## 2. 방법 1 — PydanticOutputParser (구방식, 프롬프트 주입)

### 2-1. 커스텀 ReAct 프롬프트에 Format Instructions 삽입

```python
# prompt.py — format_instructions 플레이스홀더를 추가한 커스텀 ReAct 프롬프트
REACT_PROMPT_WITH_FORMAT_INSTRUCTIONS = """Answer the following questions as best you can. \
You have access to the following tools:
{tools}

Use the following format:

Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (repeat N times)
Thought: I now know the final answer
Final Answer: the final answer formatted according to the format instructions
{format_instructions}

Question: {input}
Thought: {agent_scratchpad}"""
```

```python
# main.py — PydanticOutputParser 방식 전체 흐름
from langchain_core.output_parsers.pydantic import PydanticOutputParser
from langchain_core.prompts import PromptTemplate
from langchain_core.runnables import RunnableLambda

# 1. 출력 파서 생성 — AgentResponse 스키마 기반
output_parser = PydanticOutputParser(pydantic_object=AgentResponse)

# 2. Format Instructions를 프롬프트에 미리 주입 (partial)
react_prompt_with_format = PromptTemplate(
    template=REACT_PROMPT_WITH_FORMAT_INSTRUCTIONS,
    input_variables=["input", "agent_scratchpad", "tool_names"],
).partial(format_instructions=output_parser.get_format_instructions())

# 3. 추론 체인 생성 (기존 llm 사용)
agent = create_react_agent(llm=llm, tools=tools, prompt=react_prompt_with_format)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 4. LCEL로 파이프라인 조합
extract_output = RunnableLambda(lambda x: x["output"])          # dict → JSON 문자열
parse_output   = RunnableLambda(lambda x: output_parser.parse(x))  # JSON 문자열 → Pydantic 객체

chain = agent_executor | extract_output | parse_output

# 5. 실행 — 반환값이 AgentResponse Pydantic 객체
result: AgentResponse = chain.invoke({"input": "AI engineer jobs in Bay Area..."})
print(result.answer)
print(result.sources[0].url)
```

> 🔍 **내부 동작 원리**: `output_parser.get_format_instructions()` 가 생성하는 텍스트
>
> ```text
> "The output should be formatted as a JSON instance that conforms to the JSON schema below.
>
> As an example, for the schema {"properties": {"foo": {"title": "Foo", "description": "...",
> "type": "string"}}, "required": ["foo"]}
> the object {"foo": "bar"} is a well-formatted instance of the schema.
>
> Here is the output schema:
> {
>   "properties": {
>     "answer": {"description": "에이전트의 최종 답변", "type": "string"},
>     "sources": {"description": "...", "type": "array", "items": {...}}
>   }
> }"
>
> 이 텍스트가 프롬프트의 {format_instructions} 자리에 삽입되어 LLM에 전달됨
> LLM은 이 지침을 읽고 JSON 형식으로 Final Answer를 작성함
> output_parser.parse(json_str) → AgentResponse 객체로 변환
> ```

> ⚠️ **구방식의 문제점**: Format Instructions가 **모든 반복(iteration)** 의 프롬프트에 포함된다.
> Tool 선택 단계에서는 이 정보가 불필요하지만 매번 전송되어 **토큰 낭비**가 발생한다.
> 또한 LLM 모델 능력에 따라 형식을 어기는 경우가 발생해 파싱 에러가 나올 수 있다.

🔗 [PydanticOutputParser API 레퍼런스](https://python.langchain.com/api_reference/langchain/output_parsers/langchain.output_parsers.pydantic.PydanticOutputParser.html)

## 3. RunnableLambda와 LCEL — 체인 파이프라인 조합

**LCEL(LangChain Expression Language)** 의 `|` 파이프 연산자로 여러 단계를 선언적으로 연결한다.

```python
from langchain_core.runnables import RunnableLambda

# RunnableLambda: 일반 Python 함수를 Runnable 인터페이스로 변환
extract_output = RunnableLambda(lambda x: x["output"])
# → agent_executor 출력(dict)에서 "output" 키(JSON 문자열) 추출

parse_output = RunnableLambda(lambda x: output_parser.parse(x))
# → JSON 문자열을 AgentResponse Pydantic 객체로 변환

# | 파이프 연산자로 순차 실행
chain = agent_executor | extract_output | parse_output
```

> 🔍 **내부 동작 원리**: `|` 파이프가 실행되는 방식
>
> ```text
> chain.invoke({"input": "query"}) 호출 시:
>
>   Step 1: agent_executor.invoke({"input": "query"})
>     → {"input": "query", "output": '{"answer": "...", "sources": [...]}'}
>
>   Step 2: extract_output.invoke(step1_output)
>     → '{"answer": "...", "sources": [...]}'   (JSON 문자열)
>
>   Step 3: parse_output.invoke(step2_output)
>     → AgentResponse(answer="...", sources=[Source(url="...")])
>
> 각 Runnable은 이전 Runnable의 출력을 입력으로 받음
> RunnableLambda는 어떤 Python callable도 이 체인에 끼워 넣을 수 있게 해줌
> ```

> 💡 **실무 포인트**: `RunnableLambda`는 LCEL 체인 중간에 **커스텀 변환 로직**을 삽입하는 가장 간단한 방법이다.
> 데이터 포맷 변환, 로깅, 캐싱 등 간단한 함수를 체인에 끼워 넣을 때 활용하라.
> 단, 복잡한 로직이라면 별도 Runnable 클래스로 분리하는 것이 유지보수에 유리하다.

🔗 [RunnableLambda API 레퍼런스](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.RunnableLambda.html)  
🔗 [LCEL 개념 문서](https://python.langchain.com/docs/concepts/lcel/)

## 4. 방법 2 — `with_structured_output` (신방식, Function Calling 기반)

**현재 권장 방식**. 프롬프트 주입 없이 Function Calling을 활용해 구조화 출력을 얻는다.

```python
from langchain_openai import ChatOpenAI
from schemas import AgentResponse

llm = ChatOpenAI(model="gpt-4", temperature=0)

# structured_llm: AgentResponse 스키마에 맞는 출력만 반환하는 특수 모델 인스턴스
structured_llm = llm.with_structured_output(AgentResponse)

# 추론은 기존 llm, 마지막 구조화 단계만 structured_llm 사용
agent = create_react_agent(llm=llm, tools=tools, prompt=react_prompt)  # 일반 llm
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

extract_output = RunnableLambda(lambda x: x["output"])  # JSON 문자열 추출

# parse_output 대신 structured_llm이 마지막에 구조화
chain = agent_executor | extract_output | structured_llm

result: AgentResponse = chain.invoke({"input": "AI engineer jobs in Bay Area..."})
print(result.answer)
print(result.sources[0].url)
```

> 🔍 **내부 동작 원리**: `with_structured_output`이 하는 일
>
> ```text
> llm.with_structured_output(AgentResponse) 내부:
>   1. AgentResponse 스키마를 Function Calling용 tool 정의로 변환
>   2. 모델 호출 시 이 스키마를 API의 "tools" 파라미터로 전달
>   3. LLM은 Function Calling을 통해 스키마에 맞는 JSON을 반드시 생성
>      (프롬프트 지침이 아닌 API 레벨에서 강제)
>   4. 반환된 JSON을 LangChain이 자동으로 PydanticOutputParser로 파싱
>      → AgentResponse 인스턴스 반환
>
> LangSmith 트레이스에서 볼 수 있는 것:
>   - tools 파라미터에 AgentResponse JSON 스키마가 포함됨
>   - 매 반복이 아닌 마지막 structured_llm 호출에서만 스키마 전송
>   - 프롬프트에 format_instructions 텍스트 없음 → 토큰 절약
> ```

> ⚠️ **GPT-5 호환성 문제**: `ChatOpenAI(model="gpt-5")`로 실행 시 `stop` 파라미터 미지원으로
> `400 Bad Request` 에러 발생. 이는 ReAct 알고리즘이 `stop` 인자를 사용하기 때문이다.
> **현재 강의에서는 GPT-4를 사용**한다. Function Calling 기반 Tool Calling Agent로 전환하면
> GPT-5에서도 사용 가능하다(다음 섹션에서 다룸).

> 💡 **실무 포인트**: `create_react_agent`에는 반드시 **일반 `llm`** 을 넘겨야 한다.
> `structured_llm`을 추론 엔진에 쓰면 모든 반복에서 AgentResponse 스키마를 강제 적용하려 해
> Tool 선택 단계가 망가진다. `structured_llm`은 **마지막 출력 변환 단계에서만** 사용한다.

🔗 [with_structured_output() 공식 문서](https://python.langchain.com/docs/how_to/structured_output/)  
🔗 [Function Calling 개념 문서](https://python.langchain.com/docs/concepts/tool_calling/)

## 5. 두 방식 비교

| 항목 | PydanticOutputParser (구방식) | with_structured_output (신방식) |
|-----|----------------------------|---------------------------------|
| **구조화 메커니즘** | 프롬프트에 Format Instructions 주입 | API 레벨 Function Calling |
| **신뢰성** | 낮음 (모델 능력에 의존, 파싱 에러 가능) | 높음 (API 레벨에서 형식 강제) |
| **토큰 효율** | 낮음 (모든 반복에 스키마 포함) | 높음 (마지막 단계에서만 스키마 전송) |
| **모델 지원** | 모든 텍스트 생성 모델 지원 | Function Calling 지원 모델만 가능 |
| **코드 복잡도** | 높음 (파서 생성 + 프롬프트 수정 필요) | 낮음 (`.with_structured_output()` 한 줄) |
| **프롬프트 제어** | 높음 (직접 수정 가능) | 낮음 (LangChain이 자동 처리) |
| **현재 권장 여부** | ❌ 레거시, 학습 목적 | ✅ **프로덕션 권장** |

> 💡 **실무 포인트**: 현재 상용 모델(GPT-4/4o, Claude 3.5, Gemini 1.5 등)은 모두 Function Calling을
> 지원하므로 `with_structured_output`을 쓰지 않을 이유가 없다.
> PydanticOutputParser를 배우는 이유는 **레거시 코드 이해** 및 Function Calling 미지원 온프레미스 모델
> 사용 시 fallback 방법으로서의 가치 때문이다.

## 6. LangSmith 트레이스로 두 방식 차이 확인

**PydanticOutputParser 방식 트레이스**:

```text
AgentExecutor
  ├─ Iteration 1: [프롬프트 = ReAct + Format Instructions] → LLM → Tool 선택
  ├─ Tool 실행 (tavily_search)
  └─ Iteration 2: [프롬프트 = ReAct + Format Instructions + Observation] → LLM → Final Answer
       ↓
  RunnableLambda(extract_output): dict → JSON 문자열
  RunnableLambda(parse_output):   JSON 문자열 → AgentResponse 객체
```

**with_structured_output 방식 트레이스**:

```text
AgentExecutor
  ├─ Iteration 1: [프롬프트 = 일반 ReAct, tools에 스키마 없음] → LLM → Tool 선택
  ├─ Tool 실행 (tavily_search)
  └─ Iteration 2: [프롬프트 = 일반 ReAct + Observation] → LLM → Final Answer (텍스트)
       ↓
  RunnableLambda(extract_output): dict → 텍스트 문자열 추출
  structured_llm 호출:            [tools에 AgentResponse 스키마 포함] → JSON → AgentResponse 객체
```

> 🔍 **핵심 차이**: 신방식은 **마지막 structured_llm 호출** 하나에서만 스키마를 전송하고,
> 그 이전 모든 추론 단계에서는 깔끔한 ReAct 프롬프트만 사용한다.

## 핵심 요약

```text
✅ Pydantic BaseModel → LLM 구조화 출력의 설계도 역할
✅ Field(description="...") 품질 = LLM 출력 정확도에 직결
✅ default_factory=list → mutable 기본값의 Python 고전 버그 방지

[구방식] PydanticOutputParser
✅ get_format_instructions() → Pydantic 스키마를 텍스트로 변환해 프롬프트에 주입
✅ 모든 반복 프롬프트에 스키마 포함 → 토큰 낭비, 파싱 에러 위험
✅ Function Calling 미지원 모델에서 사용 가능한 레거시 방식

[신방식] with_structured_output
✅ Function Calling API를 통해 스키마를 강제 → 높은 신뢰성
✅ 마지막 단계에서만 structured_llm 사용 → 추론 체인에는 일반 llm 사용
✅ 코드 단순 (한 줄), 토큰 절약, 파싱 에러 없음 → 프로덕션 권장

[LCEL]
✅ | 파이프 연산자로 Runnable 순차 연결
✅ RunnableLambda → 일반 Python 함수를 Runnable로 래핑
✅ 각 단계의 입출력은 LangSmith 트레이스에서 추적 가능

[주의]
✅ create_react_agent에는 일반 llm, 마지막 파이프에만 structured_llm
✅ GPT-5는 stop 파라미터 미지원 → ReAct 방식과 호환 불가 (Tool Calling Agent 사용 필요)
```
