## 1. Document Loaders (문서 로더) — 데이터의 규격화

> **"왜 Document Loaders가 필요할까요?" (Why)**
> 현실의 데이터는 Medium 블로그, PDF 파일, YouTube 자막, Slack 대화록 등 구조와 포맷이 모두 제각각입니다. LLM은 이 복잡한 원본 데이터를 그대로 받아들일 수 없기 때문에, 모든 종류의 데이터를 읽어내어 하나의 동일한 규격(단순 텍스트 + 출처 정보)으로 평탄화(Flatten)해주는 추상화된 표준화 인터페이스가 필수적입니다.

Document Loader 모듈은 지정된 파일이나 API 경로의 데이터를 파싱(parsing)하여, LangChain의 표준 `Document` 객체 리스트로 반환합니다. 이 리스트의 각 요소는 핵심 텍스트 내용(`page_content`)과 문서의 정보(`metadata`)를 담고 있습니다.

```python
import os
from dotenv import load_dotenv
from langchain_community.document_loaders import TextLoader

load_dotenv()

# 파일 경로를 전달하여 해당 파일 내용을 Document 객체로 변환
loader = TextLoader("/Users/edenmarco/Desktop/langchain-course/mediumblog1.txt")
document = loader.load()
```

> 💡 **실무 포인트**:
>
> - **인코딩 에러 대처 방안**: 운영 체제가 사용하는 인코더 설정 때문에 파일 로드 시 `UnicodeDecodeError`가 발생하는 경우가 매우 빈번합니다. 이 때는 초기화 시 `encoding="utf-8"` 인자나 `autodetect_encoding=True` 플래그를 추가하는 것이 핵심 해결책입니다.
> - **메타데이터(Metadata)의 중요성**: `document.metadata['source']`에는 기본적으로 파일 경로가 들어갑니다. 이는 나중에 RAG 시스템이 "어떤 문서를 참고해서 답을 했는지 (Grounding)" 출처를 증명하고 데이터를 필터링할 때 없어서는 안 될 요소입니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> [다양한 텍스트 원본 소스]
>   ├─ mediumblog1.txt (TextLoader)
>   ├─ YouTube Transcript (YoutubeLoader)
>   └─ Slack Messages (SlackDirectoryLoader)
>          │
>          ▼ (load() 메서드 호출)
> ┌───────────────────────────────────────────────┐
> │ 1. 소스에 접근하여 Raw 데이터 읽기                  │
> │ 2. 파싱 엔진을 통해 순수 텍스트 추출                │
> │ 3. 텍스트는 page_content에, 파일 경로는 metadata에 매핑│
> │ 4. [Document(page_content="...", metadata=...)] 반환│
> └───────────────────────────────────────────────┘
> ```
>
> (어떤 종류의 Loader를 사용하든 `loader.load()`를 호출하면 반환되는 결과값은 모두 `Document` 리스트로 완전히 동일하게 추상화되어 있습니다. 덕분에 데이터 소스가 중간에 바뀌어도, 뒤에 이어지는 파이프라인 코드를 단 한 줄도 수정할 필요가 없습니다.)

## 2. Text Splitters (텍스트 분할기) — Context Window 한계 극복

> **"왜 텍스트 문자열을 잘게 쪼개야(Split) 할까요?" (Why)**
> LLM 모델들은 머릿속에 한 번에 담아서 연산할 수 있는 문자 길이(Context Window, Token Limit)가 엄격하게 제한되어 있습니다. 문서 전체를 LLM에 통째로 던지면 여지없이 크래시(API 에러)가 발생합니다. 또한 토큰을 많이 쓸수록 API 통신 비용이 기하급수적으로 늘어나고(Garbage in, Garbage out), 불필요한 내용이 섞이면 답변의 품질도 오히려 낮아집니다. 따라서 의미를 잃어버리지 않는 선에서 LLM이 한 입에 먹을 수 있는 크기(Chunk)로 잘라주어야 합니다.

`CharacterTextSplitter`는 로드된 전체 `Document` 객체를 지정한 글자 수(`chunk_size`)만큼 작은 청크(Chunk)로 분리합니다. 쪼개진 결과물 역시 `Document` 객체 형태를 유지합니다.

```python
from langchain_text_splitters import CharacterTextSplitter

# 1000글자(또는 토큰) 단위로 쪼개되, 청크 간 오버랩 되는 부분은 0으로 설정
text_splitter = CharacterTextSplitter(chunk_size=1000, chunk_overlap=0)

# document_loaders가 생성한 Document 객체 리스트를 분할
texts = text_splitter.split_documents(document)
print(f"Created {len(texts)} chunks")
```

> 💡 **실무 포인트**:
>
> - **Garbage In, Garbage Out**: 아무리 Gemini처럼 수백만 토큰을 처리하는 최신 모델이더라도, 질문과 관련 없는 정보까지 통째로 삼키면 노이즈 때문에 엉뚱한 결론을 도출할 확률이 높아집니다. 핵심만 간추린 작은 청크의 검색이 정확도에 훨씬 유리합니다.
> - **오버랩(Chunk Overlap)의 치명적 중요성 (위 코드에서는 0으로 실습)**: 문장이 `chunk_size`에 칼같이 맞물려 중간에서 뚝 잘려버리면 뒷부분의 문맥 연결점(Context)을 소실합니다. 이를 방지하기 위해 앞 청크의 꼬리를 뒤 청크의 머리에 약간 물리도록 중복(Overlap)을 설정해야 검색 엔진이 질문 키워드의 문맥을 완벽히 매칭할 수 있습니다.
> - **불완전한 구분자 한계**: `\n\n`과 같이 줄바꿈 문자로 스플릿하다 보면, 정확히 지정한 1000개의 문자로 안 떨어져서 `Created a chunk of size 1120...` 등의 경고 로그가 뜰 수 있습니다. 실무에서는 토큰 기반 Splitter 등 다양한 분할 전략을 조합해 사용합니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> 원본 텍스트: "안녕 나는 LLM 봇이야. LangChain은 참 재미있는 모듈이지. RAG를 배울거야."
> 
> [chunk_size=20, chunk_overlap=10 설정 시 데이터 흐름]
> 
> ┌─ chunk 1 ─────────────────────────┐
> │ "안녕 나는 LLM 봇이야. Lang"        │
> └───────────────────────────────────┘
>            ┌─ chunk 2 (10자 오버랩)─────┐
>            │ "봇이야. LangChain은 참 "  │
>            └────────────────────────────┘
> ```
>
> (`split_documents()` 메서드는 내부적으로 각 문서의 `page_content` 텍스트를 꺼내어 `separator` 단위나 글자 수 단위로 분리하고 길이 검사를 진행한 뒤 잘라낸 각 조각들을 다시 새로운 `Document` 인스턴스로 감싸 반환합니다.)

## 3. Embeddings & Vector Stores — 대규모 수학적 검색 인프라

> **"왜 임베딩과 특수 목적 DB인 Vector Store가 필요할까요?" (Why)**
> 컴퓨터는 단어의 뜻(예: "사과"와 "배"가 비슷하다는 사실)을 이해하지 못합니다. 단어의 의미적 유사도를 빠르고 완벽하게 계산하려면, 모든 텍스트 청크를 다차원 벡터 숫자 공간(Vector Space)의 수학적 좌표로 맵핑(Mapping)해야 합니다. 이렇게 수억 개의 초고차원 벡터 배열 사이에서 번개처럼 코사인 유사도 연산을 해내는(ANN 검색) 특수 목적 아키텍처가 Vector Store입니다.

`OpenAIEmbeddings` 클래스가 텍스트 데이터 덩어리를 벡터 숫자로 변환하는 역할을 담당하고, `PineconeVectorStore`가 변환된 숫자 배열들을 클라우드 DB 인덱스에 삽입(Upsert)합니다. `from_documents` 메서드는 이 두 역할을 하나로 추상화한 강력한 인터페이스입니다.

```python
from langchain_openai import OpenAIEmbeddings
from langchain_pinecone import PineconeVectorStore

# OpenAI의 기본 임베딩 모델(ada-002) 인스턴스 초기화
embeddings = OpenAIEmbeddings(openai_api_key=os.environ.get("OPENAI_API_KEY"))

print("ingesting...")
# 텍스트 청크들, 사용할 임베딩 모델, 저장할 Vector Store의 인덱스명을 넘김
PineconeVectorStore.from_documents(
    texts, 
    embeddings, 
    index_name=os.environ["INDEX_NAME"]
)
print("finish")
```

> 💡 **실무 포인트**:
>
> - **단일 인터페이스의 위력(추상화)**: 우리가 이 `for` 루프와 `API` 통신, 배치(Batch) 코드 로직을 직접 짤 수도 있지만, `PineconeVectorStore.from_documents` 한 줄을 쓰면 향후 로컬 벡터 DB(FAISS, Chroma)나 사내 호스팅 임베딩 모델로 갈아탈 때 단 두 줄의 클래스 교체만으로 완벽하게 마이그레이션이 끝납니다.
> - **프로덕션 레이트 리밋 우회 (Rate Limit)**: LangChain의 `from_documents` 메서드 내부는 이미 Async I/O(비동기 통신), 다중 스레딩, 배치 묶음(Batching) 처리를 자체적으로 지원하여 대규모 데이터를 찔러넣을 때 빈번히 터지는 API Rate Limit 한계율을 매우 효과적으로 극복하도록 설계되어 있습니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> [입력: Split 된 Document 청크 리스트 20개]
>         │
>         ▼ (PineconeVectorStore.from_documents 동작)
> ┌────────────────────────────────────────────────────────┐
> │ 1. 내부적으로 각 청크들을 순회(for loop)                     │
> │ 2. embeddings_model에게 OpenAI API 임베딩 요청 송신          │
> │    -> 응답: 20개의 1536 차원 벡터 배열 [0.12, -0.05, ...]    │
> │ 3. 속도와 Rate Limit 방어를 위해 비동기(Async) 배치로 묶음      │
> │ 4. Pinecone 벡터 DB API를 호출하여 Insert 연산 (Upsert)     │
> └────────────────────────────────────────────────────────┘
>         │
>         ▼
> [Pinecone Index 내 저장 (청크 텍스트 본문 + 메타데이터 + 1536 벡터값)]
> ```
>
> (`from_documents`와 `add_texts` 소스 코드를 들여다보면 위 과정들이 `bulk` 통신 모드로 묶여 있어 매우 촘촘한 엔지니어링 패턴으로 구현되어 있음을 알 수 있습니다.)

🔗 [Embeddings 공식 문서](https://docs.langchain.com/oss/python/integrations/text_embedding/), [Vector Stores 공식 문서](https://docs.langchain.com/oss/python/integrations/vectorstores)

## 핵심 요약

```text
✅ Document Loaders: 다양한 원본 소스에서 Raw 텍스트를 추출하여 `page_content`와 `metadata`를 가진 `Document` 규격으로 표준화.
✅ Text Splitters: LLM의 문맥 윈도우 한계를 회피하고, 연관 문서 추출 비용(비용과 노이즈)을 아끼기 위해 긴 문서를 작은 조각(Chunk) 단위로 쪼갬. Chunk Overlap 여부가 치명적이다.
✅ Embeddings & Vector Stores: 컴퓨터가 문맥 뜻을 인지할 수 있도록 자연어를 다차원 공간의 숫자 배열로 파싱. `from_documents` 등 뛰어난 추상화 인터페이스를 통해 모델 및 인프라 교체가 매우 수월하고 내부적으로 배치(Batch) 성능 최적화가 적용되어 있음.
```

## 실습 체크리스트

- [ ] 만약 `UnicodeDecodeError` 등에 직면할 경우 `encoding='utf-8'` 플래그를 추가해서 실행해결하기
- [ ] Pinecone 사이트 대시보드에서 `medium_blogs` 등 인덱스 클릭 후 생성된 20개(또는 N개)의 Vectors가 `page_content` 메타데이터 필드와 함께 잘 색인(Indexing) 되었는지 눈으로 직접 검수해보기
