## 1. Document Loaders (문서 로더) — 데이터의 규격화

> **"왜 Document Loaders가 필요할까요?" (Why)**
> 현실의 데이터는 Medium 블로그, PDF 파일, YouTube 자막, Slack 대화록 등 구조와 포맷이 모두 제각각입니다. LLM은 이 복잡한 원본 데이터를 그대로 받아들일 수 없기 때문에, 모든 종류의 데이터를 읽어내어 하나의 동일한 규격(단순 텍스트 + 출처 정보)으로 평탄화(Flatten)해주는 추상화된 표준화 인터페이스가 필수적입니다.

Document Loader 모듈은 지정된 파일이나 API 경로의 데이터를 파싱(parsing)하여, LangChain의 표준 `Document` 객체 리스트로 반환합니다. 이 리스트의 각 요소는 핵심 텍스트 내용(`page_content`)과 문서의 정보(`metadata`)를 담고 있습니다.

```python
import os
from langchain_community.document_loaders import TextLoader

# 파일 경로를 전달하여 해당 파일 내용을 Document 객체로 변환
loader = TextLoader("/Users/edenmarco/Desktop/langchain-course/mediumblog1.txt")
document = loader.load()
```

> 💡 **실무 포인트**:
>
> - **인코딩 에러 대처 방안**: 운영 체제가 사용하는 인코더 설정 때문에 파일 로드 시 `UnicodeDecodeError`가 발생하는 경우가 매우 빈번합니다. 이 때는 초기화 시 `encoding="utf-8"` 인자나 `autodetect_encoding=True` 플래그를 추가하는 것이 핵심 해결책입니다.
> - **메타데이터(Metadata)의 중요성**: `document.metadata['source']`에는 기본적으로 파일 경로가 들어갑니다. 이는 나중에 RAG 시스템이 "어떤 문서를 참고해서 답을 했는지 (Grounding)" 출처를 증명하고 데이터를 필터링할 때 없어서는 안 될 요소입니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> [다양한 텍스트 원본 소스]
>   ├─ mediumblog1.txt (TextLoader)
>   ├─ YouTube Transcript (YoutubeLoader)
>   └─ Slack Messages (SlackDirectoryLoader)
>          │
>          ▼ (load() 메서 호출)
> ┌───────────────────────────────────────────────┐
> │ 1. 소스에 접근하여 Raw 데이터 읽기                  │
> │ 2. 파싱 엔진을 통해 순수 텍스트 추출                │
> │ 3. 텍스트는 page_content에, 파일 경로는 metadata에 매핑│
> │ 4. [Document(page_content="...", metadata=...)] 반환│
> └───────────────────────────────────────────────┘
> ```
>
> (어떤 종류의 Loader를 사용하든 `loader.load()`를 호출하면 반환되는 결과값은 모두 `Document` 리스트로 완전히 동일하게 추상화되어 있습니다. 덕분에 데이터 소스가 중간에 바뀌어도, 뒤에 이어지는 파이프라인 코드를 단 한 줄도 수정할 필요가 없습니다.)

## 2. Text Splitters (텍스트 분할기) — Context Window 한계 극복

> **"왜 텍스트 문자열을 잘게 쪼개야(Split) 할까요?" (Why)**
> LLM 모델들은 머릿속에 한 번에 담아서 연산할 수 있는 문자 길이(Context Window, Token Limit)가 엄격하게 제한되어 있습니다. 문서 전체를 LLM에 통째로 던지면 여지없이 크래시(API 에러)가 발생합니다. 또한 토큰을 많이 쓸수록 API 통신 비용이 기하급수적으로 늘어나고(Garbage in, Garbage out), 불필요한 내용이 섞이면 답변의 품질도 오히려 낮아집니다. 따라서 의미를 잃어버리지 않는 선에서 LLM이 한 입에 먹을 수 있는 크기(Chunk)로 잘라주어야 합니다.

`CharacterTextSplitter`는 로드된 전체 `Document` 객체를 지정한 글자 수(`chunk_size`)만큼 작은 청크(Chunk)로 분리합니다. 쪼개진 결과물 역시 `Document` 객체 형태를 유지합니다.

```python
from langchain_text_splitters import CharacterTextSplitter

# 1000글자(또는 토큰) 단위로 쪼개되, 청크 간 오버랩 되는 부분은 0으로 설정
text_splitter = CharacterTextSplitter(chunk_size=1000, chunk_overlap=0)

# document_loaders가 생성한 Document 객체 리스트를 분할
texts = text_splitter.split_documents(document)
print(f"Created {len(texts)} chunks")
```

> 💡 **실무 포인트**:
>
> - **Garbage In, Garbage Out**: 아무리 Gemini처럼 수백만 토큰을 처리하는 최신 모델이더라도, 질문과 관련 없는 정보까지 통째로 삼키면 노이즈 때문에 엉뚱한 결론을 도출할 확률이 높아집니다. 핵심만 간추린 작은 청크의 검색이 정확도에 훨씬 유리합니다.
> - **오버랩(Chunk Overlap)의 치명적 중요성 (위 코드에서는 0으로 실습)**: 문장이 `chunk_size`에 칼같이 맞물려 중간에서 뚝 잘려버리면 뒷부분의 문맥 연결점(Context)을 소실합니다. 이를 방지하기 위해 앞 청크의 꼬리를 뒤 청크의 머리에 약간 물리도록 중복(Overlap)을 설정해야 검색 엔진이 질문 키워드의 문맥을 완벽히 매칭할 수 있습니다.
> - **불완전한 구분자 한계**: `\n\n`과 같이 줄바꿈 문자로 스플릿하다 보면, 정확히 지정한 1000개의 문자로 안 떨어져서 `Created a chunk of size 1120...` 등의 경고 로그가 뜰 수 있습니다. 실무에서는 토큰 기반 Splitter 등 다양한 분할 전략을 조합해 사용합니다.

> 🔍 **내부 동작 원리**:
>
> ```text
> 원본 텍스트: "안녕 나는 LLM 봇이야. LangChain은 참 재미있는 모듈이지. RAG를 배울거야."
> 
> [chunk_size=20, chunk_overlap=10 설정 시 데이터 흐름]
> 
> ┌─ chunk 1 ─────────────────────────┐
> │ "안녕 나는 LLM 봇이야. Lang"        │
> └───────────────────────────────────┘
>            ┌─ chunk 2 (10자 오버랩)─────┐
>            │ "봇이야. LangChain은 참 "  │
>            └────────────────────────────┘
> ```
>
> (`split_documents()` 메서드는 내부적으로 각 문서의 `page_content` 텍스트를 꺼내어 `separator` 단위나 글자 수 단위로 분리하고 길이 검사를 진행한 뒤 잘라낸 각 조각들을 다시 새로운 `Document` 인스턴스로 감싸 반환합니다.)

## 핵심 요약

```text
✅ Document Loaders: 다양한 원본 소스에서 Raw 텍스트를 추출하여 `page_content`와 `metadata`를 가진 `Document` 규격으로 표준화합니다. 추상화 인터페이스를 통해 백엔드 로더 코드가 변경되어도 이후 로직은 유지됩니다.
✅ Text Splitters: LLM의 문맥 윈도우 한계를 회피하고, 연관 문서 추출 시 Garbage In, Garbage Out 비용 낭비를 막기 위해 긴 문서를 작은 조각(Chunk) 단위로 분할합니다.
✅ Chunk Overlap: 문턱 기반 분할에서 일어나는 문맥 단절 현상을 방지하는 RAG 검색 성능의 핵심 튜닝 옵션입니다.
```
